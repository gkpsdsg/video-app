## #1 — backend: ranking.service.ts 全量加载 + PostgreSQL 崩溃

**日期：** 2026-05-22

**现象：**
- `/feed/hot` 接口响应极慢，随数据量增长越来越慢
- PostgreSQL 报错：`DISTINCT + RANDOM()` 在有大表 JOIN 时崩溃

**根因：**
1. `findAll()` 没有分页，一次性把全部视频加载到内存再排序
2. PostgreSQL 不支持 `DISTINCT ON` + `ORDER BY RANDOM()` 直接组合，导致数据库层报错

**修复：**
- `findAll()` 改用 Redis ZSET 做热度排序，只查当前页需要的视频
- 随机选取改为两步：先在子查询取 ID，再在主查询用 `WHERE id IN (...)` 获取详情
- 添加分页参数 `page` / `limit`

**涉及文件：** `backend/src/ranking/ranking.service.ts`

## #2 — frontend: TextEditingController dispose 后使用

**日期：** 2026-05-22

**现象：**
- 评论底部弹窗关闭后报异常：`A TextEditingController was used after being disposed`

**修复：**
- 在异步回调中加 `mounted` 检查后再访问 controller
- 提交评论后不重置 controller 文本（改用新 controller 实例）

**涉及文件：** `frontend/lib/screens/feed_screen.dart`

**日期：** 2026-05-22

**现象：**
- 视频 feed 页底部导航栏区域报 RenderFlex 溢出，溢出量 99715px

- 视频播放器用 `AspectRatio` 包裹，确保不超出屏幕

## #9 — Android release APK 无法网络请求（INTERNET 权限缺失）

**日期：** 2026-05-24

**现象：**
- 手机安装 release APK 后，登录完全无反应
- 抓包工具显示没有任何网络数据包发出
- 手机浏览器访问同一 URL 正常

**根因：**
- Flutter 模板只在 `debug/AndroidManifest.xml` 和 `profile/AndroidManifest.xml` 中添加了 `<uses-permission android:name="android.permission.INTERNET"/>`
- `main/AndroidManifest.xml`（release 构建使用）**没有** INTERNET 权限
- 无网络权限 → 所有 HTTP 请求被系统静默丢弃 → Dio 报 `connectionError`

**修复：**
- `main/AndroidManifest.xml` 添加 `<uses-permission android:name="android.permission.INTERNET"/>`

**教训：**
- debug/profile/release 三个 manifest 的差异极易被忽略
- 新建 Flutter 项目后应立即检查 release manifest 的关键权限
- 添加网络请求后应第一时间用 release 包测试，而非等到最后

**涉及文件：** `frontend/android/app/src/main/AndroidManifest.xml`

## #10 — Android 9+ HTTP 明文请求被拦截

**日期：** 2026-05-23

**现象：**
- APK 登录点按钮后无反应，抓包工具抓不到任何数据包
- 后端日志无任何请求到达

**根因：**
- Android 9（API 28）起默认禁止 HTTP 明文流量
- APK 硬编码的是 `http://10.198.238.77:3000`（HTTP 非 HTTPS）
- 缺少 `usesCleartextTraffic` 声明 → 系统在 TCP 层之前就拦截了请求

**修复：**
- AndroidManifest.xml 添加 `android:usesCleartextTraffic="true"`
- 创建 `network_security_config.xml`，`base-config` 中声明 `cleartextTrafficPermitted="true"`

**涉及文件：**
- `frontend/android/app/src/main/AndroidManifest.xml`
- `frontend/android/app/src/main/res/xml/network_security_config.xml`（新建）

## #11 — 真机网络连通：逐层排查记录

**日期：** 2026-05-23 ~ 2026-05-24

**背景：**
后端部署在 VMware 虚拟机（192.168.161.128），需从物理宿主机 WiFi 热点对外提供服务。用户多次换网（校园 WiFi / 手机热点），出现过以下全部问题。

### 排查链路

```
手机 (WiFi)
  → 宿主机 WiFi (10.198.238.77)
    → 宿主机 VMnet8 (192.168.161.1)
      → 虚拟机 Ethernet0 (192.168.161.128)
        → NestJS :3000 / MinIO :9000
```

### 子问题 A：VMnet8 虚拟网卡 IP 丢失

**现象：** 宿主机 `curl http://192.168.161.128:3000/` 超时

**诊断：**
```cmd
ipconfig
```
VMware Network Adapter VMnet8 显示 APIPA 地址 `169.254.x.x` 而非 `192.168.161.x`

**根因：** VMware NAT 网络的 DHCP 分配失败，或手动 IP 在重启后丢失

**修复：**
```cmd
netsh interface ip set address "VMware Network Adapter VMnet8" static 192.168.161.1 255.255.255.0
```

**验证：** `curl http://192.168.161.128:3000/` → "Hello World!"

---

### 子问题 B：端口转发规则重启后丢失

**现象：** 宿主机 `curl http://localhost:3000/` 无法连接

**诊断：**
```cmd
netsh interface portproxy show all
```
输出为空

**根因：** Windows `netsh portproxy` 规则不会持久化到重启后

**修复：**
```cmd
netsh interface portproxy add v4tov4 listenport=3000 listenaddress=0.0.0.0 connectport=3000 connectaddress=192.168.161.128
netsh interface portproxy add v4tov4 listenport=9000 listenaddress=0.0.0.0 connectport=9000 connectaddress=192.168.161.128
```

**教训：** 应创建启动脚本或 Windows 计划任务在开机时自动添加上述规则

---

### 子问题 C：防火墙规则重启后丢失

**现象：** 端口转发恢复后，手机仍无法访问 `http://10.198.238.77:3000`

**诊断：**
```cmd
netsh advfirewall firewall show rule name="Port 3000"
```
→ "No rules match the specified criteria"

**根因：** Windows 防火墙规则同样不会自动持久化（取决于添加方式）

**修复：**
```cmd
netsh advfirewall firewall add rule name="Port 3000" dir=in action=allow protocol=TCP localport=3000
netsh advfirewall firewall add rule name="Port 9000" dir=in action=allow protocol=TCP localport=9000
```

---

### 子问题 D：宿主机无法 curl 自己的外部 IP

**现象：** 宿主机 `curl http://10.198.238.77:3000/` 失败，但 `curl localhost:3000` 正常

**诊断：** 这是 Windows 的预期行为——Windows 不支持 hairpin NAT（回环 NAT），主机无法通过自己的外部 IP 访问自己

**结论：** 不是问题，只需要在手机端测试

---

### 子问题 E：手机连不同热点时 IP 变化

**现象：** 用户在校园 WiFi（172.x.x.x）和手机热点（10.198.x.x）之间切换，每次 IP 都变

**影响：**
- APK 中硬编码的 API 地址失效
- MinIO 预签名 URL 中的 endpoint 失效
- 需要重新打包 APK 或修改 `.env`

**缓解措施：** 确认一个稳定的网络环境后保持不变

## #12 — 视频加载不出（MinIO 预签名 URL 内网地址）

**日期：** 2026-05-24

**现象：**
- 登录成功，API 返回视频元数据正常
- 视频播放器黑屏，加载不出来

**诊断过程：**
1. 后端日志显示所有 API 查询正常 → 排除接口问题
2. 检查 `/video/:id/stream` 返回值 → URL 是 `http://localhost:9000/...`
3. `localhost:9000` 在手机上是手机自己，不是服务器 → 确认根因

**根因：**
- `.env` 中 `MINIO_ENDPOINT=localhost`
- MinIO JS SDK 用此 endpoint 生成预签名 URL
- 手机拿到 URL 后访问 `localhost:9000` 指向的是自己，而非服务器

**修复：**
1. 改 `MINIO_ENDPOINT=10.198.238.77`（宿主机外部 IP）
2. 宿主机添加端口转发：`0.0.0.0:9000 → 192.168.161.128:9000`
3. 宿主机添加防火墙规则允许 TCP 9000
4. 重启后端使配置生效

**涉及文件：** `backend/.env`

## #13 — 登录失败无错误提示

**日期：** 2026-05-24

**现象：**
- 登录失败时，按钮显示 loading 几秒后恢复，但没有任何错误提示
- 用户不知道是密码错了、网络不通、还是其他原因

**根因：**
- `AuthProvider.login()` 用 `try/finally` 而非 `try/catch`——异常被静默吞掉
- `LoginScreen._submit()` 只检查 `auth.isLoggedIn`，不检查是否有错误

**修复：**
- `AuthProvider` 新增 `error` 属性和 `_extractError()` 方法，分类捕获 `DioException`：
  - `connectionTimeout` / `receiveTimeout` → "连接超时，请检查网络"
  - `connectionError` → "无法连接服务器，请检查网络设置"
  - `badResponse` → 返回服务端错误消息或状态码
  - 其他 → "网络错误，请稍后重试"
- `LoginScreen._submit()` 在失败时显示红色 SnackBar 提示

**涉及文件：**
- `frontend/lib/providers/auth_provider.dart`
- `frontend/lib/screens/login_screen.dart`

## #14 — 切换 WiFi 后视频加载不出（完整排查清单）

**日期：** 2026-05-25

**现象：**
- 校园网 IP 从 `172.29.255.221` 变为 `172.28.25.193`
- 手机端登录成功，但视频全部加载不出来（黑屏/转圈）

**根因：**
切换 WiFi 导致宿主机 IP 变化后，**4 个地方**需要同步修改，漏一个就挂：

| # | 位置 | 作用 | 漏改后果 |
|---|------|------|----------|
| 1 | `frontend/lib/main.dart` | 手机访问后端的地址 | 手机连不上 API |
| 2 | `backend/.env` → `MINIO_ENDPOINT` | MinIO 预签名 URL 的 host | 手机拿到 `localhost` 或旧 IP 的 URL，视频加载失败 |
| 3 | 宿主机 `portproxy` | 转发宿主机端口 → 虚拟机 | 请求到不了虚拟机 |
| 4 | 宿主机防火墙 | 放行 3000 / 9000 端口 | 请求被 Windows 防火墙拦截 |

本次同时触发了上述全部问题：校园网 IP 变了，`portproxy` 和防火墙规则之前重启后也丢了。

**诊断步骤（按顺序排查）：**

```cmd
# 1. 确认宿主机当前 WiFi IP
ipconfig | findstr "WLAN"

# 2. 确认 portproxy 规则存在
netsh interface portproxy show all

# 3. 确认防火墙规则存在
netsh advfirewall firewall show rule name="Port 3000"
netsh advfirewall firewall show rule name="Port 9000"

# 4. 从宿主机验证可以到达虚拟机
curl http://192.168.161.128:3000/

# 5. 从手机浏览器访问 http://<宿主机IP>:3000/ 验证连通
```

**重启后必做清单（管理员 PowerShell）：**

> Windows 的 `netsh portproxy` 和 `netsh advfirewall` 规则**重启后全部丢失**。每次重启电脑后都要执行以下 4 条命令：

```cmd
netsh interface portproxy add v4tov4 listenport=3000 listenaddress=<当前IP> connectport=3000 connectaddress=192.168.161.128
netsh interface portproxy add v4tov4 listenport=9000 listenaddress=<当前IP> connectport=9000 connectaddress=192.168.161.128
netsh advfirewall firewall add rule name="Port 3000" dir=in action=allow protocol=TCP localport=3000
netsh advfirewall firewall add rule name="Port 9000" dir=in action=allow protocol=TCP localport=9000
```

**修复（宿主机管理员 PowerShell）：**

```cmd
# 端口转发（注意 listenaddress 用当前 WiFi IP，不能用 0.0.0.0）
netsh interface portproxy add v4tov4 listenport=3000 listenaddress=<当前IP> connectport=3000 connectaddress=192.168.161.128
netsh interface portproxy add v4tov4 listenport=9000 listenaddress=<当前IP> connectport=9000 connectaddress=192.168.161.128

# 防火墙
netsh advfirewall firewall add rule name="Port 3000" dir=in action=allow protocol=TCP localport=3000
netsh advfirewall firewall add rule name="Port 9000" dir=in action=allow protocol=TCP localport=9000
```

然后修改代码中的 IP：
- `frontend/lib/main.dart` → 新 IP
- `backend/.env` → `MINIO_ENDPOINT=<新IP>`
- 重新打包 APK，重启后端

**涉及文件：** `frontend/lib/main.dart`、`backend/.env`、宿主机 portproxy + 防火墙

## 教训总结

1. **release 构建要早测** — INTERNET 权限、明文流量限制等问题在 debug 构建中不暴露
2. **Windows 网络规则不持久** — `portproxy`、防火墙、静态 IP 在重启后全部丢失，应自动化
3. **内网地址绝不返回给客户端** — MinIO 预签名 URL 的 endpoint 必须是客户端可访问的外部地址
4. **错误信息要可见** — 静默失败是最差的用户体验，至少让用户知道出了什么问题
5. **逐层排查 > 乱猜** — 网络问题按 OSI 层逐级验证：进程 → 本机 → 内网 → 公网，每一步 curl 验证
